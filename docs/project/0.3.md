# Compiler project 0.3

Project `0.3` adds an explicit H.264 encoding policy to every rendition while
retaining the strict state, timing, sizing, and source contracts from project
`0.2`. Compilation always decodes the author-supplied source and encodes
independently decodable AVC units; source-video passthrough is not supported.

Use the highest-quality available source. Preferred opaque masters include
ProRes 422 HQ, DNxHR HQX, FFV1, or an RGBA PNG sequence. Preferred alpha
masters include ProRes 4444 or an RGBA PNG sequence. H.264, HEVC, AV1, and VP9
inputs can be decoded when the caller's FFmpeg supports them, but they are
delivery codecs and will be transcoded again. Metadata cannot prove that an
input is an original master.

## Rendition encoding

Each rendition selects one allowlisted libx264 preset and either capped CRF or
average-bitrate rate control.

```json
{
  "id": "avc.1x",
  "width": 1280,
  "height": 720,
  "encoding": {
    "codec": "h264",
    "preset": "slow",
    "rateControl": {
      "mode": "crf",
      "crf": 20,
      "maxBitrate": 10000000
    }
  }
}
```

Allowed presets are `ultrafast`, `superfast`, `veryfast`, `faster`, `fast`,
`medium`, `slow`, `slower`, and `veryslow`. CRF must be an integer from `1`
through `51`. Lower CRF values normally produce larger, higher-quality H.264;
x265 and x264 CRF numbers are not equivalent. For visual review, start around
x264 CRF `18` through `23` and measure the resulting asset. Packed-alpha
renditions are judged after decoding and may require a much lower CRF than an
opaque rendition; the compiler rejects output that exceeds its alpha or
composite-error limits instead of silently accepting damaged transparency.

ABR uses this shape:

```json
{
  "encoding": {
    "codec": "h264",
    "preset": "slow",
    "rateControl": {
      "mode": "abr",
      "averageBitrate": 6000000,
      "maxBitrate": 10000000
    }
  }
}
```

The compiler records the configured policy, canonical encoded bytes, and
measured average bitrate in the build report. It rejects output whose measured
average exceeds the declared ceiling.

## Compiler-owned constraints

Encoding policy is structured; arbitrary FFmpeg arguments are never accepted.
The compiler owns H.264 Baseline/CAVLC, yuv420p, BT.709 limited range, closed
GOPs, one reference frame, no B-frames, repeated parameter sets, one slice,
deterministic threads, audio removal, and raw Annex-B output. These constraints
make every authored unit independently decodable by the browser runtime.

`faststart` and `hvc1` do not apply: an AVAL asset is not an MP4 file. HEVC, VP9, and
AV1 output require separate AVAL profiles, bitstream validators, browser-runtime
support, and certification. WebM is a Matroska-derived container rather than a
video codec, so placing a WebM file inside an AVAL asset would duplicate container/index
metadata; a future open-codec path should define VP9 or AV1 AVAL sample profiles
instead. `.webm` is also not currently an accepted source-container extension.
